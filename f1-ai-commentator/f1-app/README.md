# F1 Real-Time Telemetry + Historical-Context Commentary

Turn the raw firehose of Formula 1 telemetry into sparse, meaningful, human-readable
commentary — grounded in each driver's real history at the circuit, and generated by
an LLM only when something actually worth saying happens.

> [Live demo](https://cerulean-twilight-aba1d9.netlify.app/): a React dashboard streams a replayed race (2023 Dutch GP, a wet-to-dry
> thriller) with standings, live telemetry, and an AI commentator that starts talking
> from the chaotic opening laps.


---

## 1. The problem we're solving

A modern F1 car emits **thousands of data points per second** — lap times, sector
splits, tyre compounds and age, gaps, pit-lane times, track and air temperature. Two
problems follow from that firehose:

- **Raw numbers are meaningless to most people.** "Sector 2: 41,910 ms" tells a viewer
  nothing. *Is that fast? Slow? A mistake? A tyre falling off?* The number only acquires
  meaning **in context** — compared to that driver's own best here, to the field, to
  history.
- **No human can hold all that context for 20 cars at once.** A great commentator adds
  exactly this context ("that's his quickest lap here in three years"), but a person
  can't simultaneously recall every driver's historical benchmarks at every circuit and
  apply them lap-by-lap across the whole grid.

Existing telemetry dashboards show the numbers; they don't tell you **what matters or
why**. The gap is the layer in between: automatically deciding *which* moments are
noteworthy, and explaining them *in plain language, from data*.

This project fills that gap.

---

## 2. Our AI / technical approach

The core idea is a **two-stage pipeline that separates "what to say" from "how to say
it"** — and grounds the language model in real data so it informs rather than
hallucinates.

```
 FastF1 race replay ──► Trigger engine ──► Context assembler ──► LLM (IBM Granite) ──► Dashboard
   (telemetry stream)   (WHAT matters)     (retrieve history)    (HOW to say it)     (WebSocket)
        per lap            ~4.5% fire        SQLite lookup        Langflow/Ollama      live feed
```

### Stage 1 — A deterministic trigger engine decides *what's worth talking about*

Calling an LLM on every lap of every car (~1,400 events per race) would be expensive,
slow, and — worse — **noise**. Instead, pure, side-effect-free functions evaluate each
new lap against a **rolling 5-lap buffer** and **historical benchmarks**:

| Trigger | Fires when |
|---|---|
| `fast_lap` / `slow_lap` | lap is >8% faster / >20% slower than the driver's rolling average |
| `sector_outlier` | a sector is >45% over the driver's own best at this track |
| `pit_anomaly` | a pit stop is well off the driver's historical average here |
| `drs_battle` | DRS open and the gap ahead is under 1 second |

Thresholds were **calibrated against the real distribution** of a full 20-car race, not
guessed: a naïve "5% off best" cut-point fired on ~61% of laps because race pace runs
far off ultimate pace. The tuned cut-points sit in the **tail** of the distribution, so
only genuinely unusual events fire — about **64 commentary moments across a 78-lap race
(~4.5% of driver-laps)**. That's broadcast-realistic: roughly one remark every lap or
two, concentrated exactly where a human would also suddenly have a lot to say.

### Stage 2 — Retrieval-augmented generation grounds the commentary in history

When a trigger fires, a **context assembler** queries a local SQLite database of
**2022–2024 historical results** (64k+ laps, 2.6k pit stops) and enriches the live
snapshot with that driver's **all-time best, historical average, and prior pit stops at
this circuit**. The enriched packet —

```json
{ "trigger": "slow_lap",
  "snapshot": { "driver": "OCO", "lap_time_ms": 101755, "compound": "INTERMEDIATE", ... },
  "context":  { "driver_historical_avg_ms": 84769, "total_historical_laps": 127, ... } }
```

— is sent to **IBM Granite** (running locally via **Ollama**, behind a **Langflow**
flow) which returns one grounded sentence:

> *"In lap 56, Ocon, on intermediate tyres, recorded a slow lap of 101.8s — 23.7% off his
> historical average here."*

This is **RAG for racing**: structured F1 history is the retrieval store, injected per
event so the model speaks **from data, not from its imagination**. The LLM sits behind a
single swappable seam (Langflow), so Granite-via-Ollama (local, free) can be exchanged
for IBM watsonx.ai (cloud) without touching the pipeline.

### Engineering that makes it real-time

- **FastAPI + WebSocket** backend streams telemetry to all clients every lap; commentary
  is generated as a fire-and-forget task on a dedicated thread pool so the 1–3 s LLM call
  **never stalls the telemetry stream** ("commentary a beat late is fine").
- **FastF1** provides verified, cached race data (no scraping); all times normalised to
  integer milliseconds in SQLite.
- **React + Vite + Tailwind** dashboard: standings, a live-lap card, and a scrolling,
  team-colour-coded commentary feed.

## 3. Why this matters in the context of racing

Racing is, fundamentally, a sport of **comparison** — against your own best, against your
rivals, against the history of the track. Raw telemetry alone throws that away. Adding
the historical layer back, automatically and in real time, is where the value is:

- **It democratizes understanding.** A casual fan doesn't know that 1:41.9 is dreadful or
  brilliant. "23.7% off his average here, struggling on the inters" they understand
  instantly. The system turns a wall of numbers into a story anyone can follow.
- **It scales attention beyond what humans can.** Broadcast commentary necessarily
  follows the leaders and the obvious battles. This engine watches **all 20 cars with
  full historical context simultaneously**, surfacing the midfield sector that just fell
  off a cliff or the rookie's unexpectedly strong stint — moments a human team would miss.
- **The same engine generalises to the pit wall.** "This sector is 45% over baseline" is
  exactly the kind of anomaly a **race engineer** cares about — tyre degradation, a
  setup problem, traffic, a mistake. The trigger + context layer is a strategy-and-
  engineering tool wearing a commentary hat.
- **It's robust by design.** Commentary only fires on real events, so it stays signal,
  not noise; the LLM is grounded in data, so it stays accurate, not hallucinated; and a
  failing/slow LLM degrades gracefully — telemetry keeps streaming regardless.

In short: F1 already has the data and the drama. This project supplies the missing
**interpretation layer** — deciding what's noteworthy and explaining it from the record —
making the sport's richest signal legible, at the scale of the whole grid, in real time.

## Tech stack

| Layer | Tech |
|---|---|
| Data | FastF1 (cached 2022–2024), SQLite |
| Backend | Python 3.14, FastAPI, Uvicorn, WebSockets, Pydantic v2 |
| Trigger / context | Pure-function trigger engine, SQLite-backed RAG context |
| LLM | IBM Granite via Ollama (local) behind a Langflow flow (swappable for watsonx.ai) |
| Frontend | React 18, Vite, Tailwind CSS |
| Delivery | Netlify (static dashboard) + ngrok tunnel to the local backend |

## Architecture & data flow

```
backend/
  history.py        FastF1 loader + SQLite query helpers (the retrieval store)
  db.py             SQLite schema
  telemetry.py      ReplayPoller — replays a cached race lap-by-lap, rolling buffer
  snapshot.py       Pydantic 12-field telemetry contract
  triggers.py       Pure trigger functions + evaluate() (Stage 1: what matters)
  commentary.py     assemble() — enriches a trigger with historical context (Stage 2: RAG)
  langflow_client.py POSTs the enriched packet to Langflow/Granite, extracts the text
  main.py           FastAPI server: replay loop, WebSocket broadcast, REST, /api/reset
  run_replay.py     CLI entrypoint (standalone, no server)
frontend/           React dashboard (standings, telemetry, commentary feed)
```

WebSocket message contract: `TELEMETRY_UPDATE`, `COMMENTARY_UPDATE`, `RACE_FINISHED`.
REST: `/api/status`, `/api/standings`, `/api/history/{driver}`, `POST /api/reset`.

## Running it

Prerequisites: the Python venv, Ollama with `granite3.3` pulled, and Langflow running
your flow (config in `.env`). The SQLite DB and FastF1 cache ship ready to use.

```bash
# 1. Ollama (the LLM)
ollama serve                      # ensure: ollama pull granite3.3

# 2. Langflow (the flow wrapping Granite) — running on the URL in .env

# 3. Backend
cd backend && python -m uvicorn main:app --port 8000
#   tip: REPLAY_TICK_S=0.3 for a fast demo

# 4. Frontend
cd frontend && npm install && npm run dev      # http://localhost:3000
```

Pick the race with `REPLAY_YEAR` / `REPLAY_ROUND` in `.env` (default: 2023 Dutch GP, a
wet-to-dry race whose triggers fire from the opening laps). The **RESET RACE** button in
the dashboard restarts the replay from lap 1 via `POST /api/reset`.
